# Skill S-17: Parallel Team Orchestration (병렬 팀 조율)

> **스킬 유형**: 운영·품질
> **핵심**: 에이전트를 병렬로 호출하는 표준 패턴
> **사용 에이전트**: T19 Harness Librarian

## 핵심 원칙

> **의존성이 없으면 반드시 병렬 호출.**
> 직렬은 출력이 다음의 입력일 때만.

## 의존성 분석 결정 트리

```
질문 1: 한 에이전트의 출력이 다른 에이전트의 입력인가?
  ├── 예 → 직렬 (선행 → 후행)
  └── 아니오 → 다음

질문 2: 같은 정보를 다른 관점에서 보는가?
  ├── 예 → 병렬 (모두에게 같은 컨텍스트, 각자 의견)
  └── 다음

질문 3: 같은 단계(STEP)에 속하는가?
  ├── 예 → 병렬
  └── 아니오 → STEP별 직렬, STEP 내 병렬
```

## 회의 단계 표준 (모두 병렬)

회의 단계에서는 **모든 참여자가 동시에 의견 제시**.

```typescript
// ✅ 올바른 호출 — 단일 메시지에 여러 Agent
[
  Agent(R1_CTO, "기술 결정 사인오프"),
  Agent(R2_PM, "우선순위 결정"),
  Agent(T2_ItineraryGraph, "DAG 구조 설계"),
  Agent(T5_LiveReplan, "옵션 생성 알고리즘"),
  Agent(T12_QA, "테스트 시나리오 사전 검토"),
] in single message

// ❌ 잘못된 호출 — 직렬
Agent(R1_CTO) → wait → Agent(R2_PM) → wait → ...
```

## 단계별 병렬/직렬 매핑

| 단계 | 호출 방식 | 이유 |
|------|----------|------|
| STEP 1 Triage | 단일 (T19) | 검색은 한 번에 |
| STEP 2 회의 | **병렬 N개** | 의견 수집 |
| STEP 3 구현 | 단일 또는 짝 | 일관성 보장 |
| STEP 4 검증 ① | 단일 (구현자 자체) | |
| STEP 4 검증 ② | 단일 (T13) | 차수별 진행 |
| STEP 4 검증 ③ | 단일 (T12) | 시나리오별 진행 |
| STEP 4 검증 ④ | 단일 (R1) | 최종 사인오프 |
| STEP 5 보고 | 단일 (T18) | 종합 |

> **검증은 직렬**이지만 **각 검증 내부**는 병렬 가능 (예: T13의 1차/2차/3차).

## 병렬 호출 산출물 통합

각 에이전트의 출력을 **통합 보고서**로 합침:

```markdown
## 회의 결과 — [작업명]

### R1 CTO 의견
- 핵심: ...
- 게이트: 통과 / 보류 / 반려

### R2 PM 의견
- 우선순위: ...

### T2 의견
- DAG 설계: ...

### T5 의견
- 옵션 생성: ...

### T12 의견
- 테스트 시나리오: ...

### 합의 사항
1. ...
2. ...

### 미해결 이슈
1. ... → [담당 에이전트] 재검토
```

## 병렬의 한계

다음은 **병렬로 하면 안 됨**:

| 상황 | 이유 |
|------|------|
| 자원 락 (DB 마이그레이션) | 순서 보장 필요 |
| 검증 차수 (1차→2차→3차) | 이전 단계 통과 전제 |
| CTO 사인오프 후 행동 | 사인오프는 게이트 |
| 사용자 결정 대기 | 사람의 입력 필요 |

## 회의 멤버 표준 (CATALOG에서 가져옴)

CATALOG의 "작업 유형별 책 묶음" 표 사용. T19는 이 표를 참조하여 병렬 호출.

## 호출 도구 매핑

| 호출 종류 | 도구 |
|----------|------|
| 의견 수집 (가벼움) | 인용/요약 (별도 Agent 도구 안 씀) |
| 조사 (검색·분석) | `Agent` (subagent) 병렬 |
| 검증 (테스트) | `Bash` (`npm test`) 단일 |
| 메모리 저장 | `Write` 단일 |

## 효율 측정

| 지표 | 목표 |
|------|------|
| 회의 단계 1회 라운드 시간 | < 30초 (병렬 덕분) |
| 직렬 호출 비율 | < 20% |
| 미해결 이슈로 재호출 | < 10% |

## 안티 패턴

| 안티 패턴 | 대응 |
|----------|------|
| 모두 직렬 호출 (병렬 활용 X) | 의존성 분석 → 병렬화 |
| 너무 많은 병렬 (10명+) | 표준 매핑 슬림화 |
| 병렬 후 통합 누락 | 통합 보고서 표준 적용 |
