# Skill S-18: CTO Review Gate (CTO 게이트) ⭐ 절대 규칙

> **스킬 유형**: 운영·품질 (절대 규칙)
> **핵심**: 모든 기술 결정에 R1 CTO의 명시적 사인오프
> **사용 에이전트**: R1 CTO, T19 Harness Librarian (게이트 트리거)

## 절대 원칙

> **CTO 사인오프 없이 기술 결정 금지.**
> 일단 코딩하고 나중에 통보 → 금지.
> CTO가 "잠깐"이라고 하면 모든 작업 정지.

## 게이트 발동 트리거

다음 중 하나라도 해당하면 **R1 CTO 호출 필수**:

- [ ] 메이저 버전 업그레이드
- [ ] 의존성 제거

### 2. 데이터 변경

- [ ] DB 인덱스 추가/제거
- [ ] 데이터 마이그레이션 (백필)

### 3. 외부 API
- [ ] 새 외부 API 도입 (Google/Naver/OTA/LLM 등)
- [ ] API 호출 비용 구조 변경
- [ ] Rate limit 정책 변경

### 4. 보안/인증
- [ ] 인증 메커니즘 변경
- [ ] 권한 모델 변경

### 6. 핵심 알고리즘

## 게이트 통과 절차

```
1. 변경 제안자가 ADR 초안 작성 (3~5단락)
        ↓
2. R1 CTO에게 회의 단계에서 호출 (병렬 가능)
        ↓
3. CTO 검토:
   - 아키텍처 일관성
   - 보안/프라이버시 영향
   - 성능 영향
   - 비용 영향
   - 기술 부채
        ↓
4. CTO 결정:
   - ✅ 승인 → ADR 확정 + 구현 진행
   - ⚠️ 조건부 승인 → 조건 충족 후 진행
   - ❌ 반려 → 대안 검토
        ↓
5. ADR을 docs/adr/에 저장
```

## ADR 템플릿

```markdown
# ADR-NNN: [결정 제목]

**상태**: Proposed | Accepted | Rejected | Superseded
**날짜**: YYYY-MM-DD
**결정권자**: R1 CTO
**제안자**: [에이전트 ID]

## 컨텍스트
- 현재 상황은 무엇인가?
- 왜 결정이 필요한가?

## 결정
- 무엇을 하기로 했는가?

## 대안
- 검토한 대안 1: ... (장단점)
- 검토한 대안 2: ... (장단점)
- 채택한 안: ... (이유)

## 영향
- 긍정적: ...
- 부정적: ...
- 트레이드오프: ...

## 관련
- 영향받는 스킬: S-XX
- 영향받는 에이전트: T-XX
- 후속 작업: ...
```

## 게이트 실패 시 대응

| 결정 | 대응 |
|------|------|
| ⚠️ 조건부 승인 | 조건 명시 → 후속 점검 일정 |
| ❌ 반려 | 대안 검토 회의 재개 |
| 보류 | 추가 정보 수집 후 재상정 |

## 안티 패턴 (절대 금지)

| 안티 패턴 | 신호 |
|----------|------|
| **사후 통보** | "이미 npm install 했어요, 이제 검토해 주세요" |
| **CTO 우회** | "PM이 OK 했으니까 진행" |
| **암묵적 변경** | 디자인 토큰 슬쩍 수정 |
| **의존성 폭주** | 한 PR에 3개 이상 새 라이브러리 |

> 위반 시 T13 코드 리뷰에서 즉시 반려.

## 게이트 효율 (CTO 부담 최소화)

| 정책 | 이유 |
|------|------|
| 표준 패턴 사전 승인 (ADR-001~008) | 매번 묻지 않음 |
| 병렬 회의 호출 | CTO가 다른 의견과 동시 검토 |
| ADR 초안 사전 작성 | 즉답 가능 |
| 승인 결과 캐시 (12개월) | 같은 결정 재발 방지 |

## 측정 지표

| 지표 | 목표 |
|------|------|
| 게이트 통과율 (1차 승인) | > 70% |
| ADR 작성 시간 | < 30분 |
| 게이트 우회 시도 | 0 |
| 사후 발견된 미통과 결정 | 0 |
