# Skill S-15: Code Review Checklist (코드 리뷰 체크리스트)

> **스킬 유형**: 운영·품질
> **핵심**: 다중 리뷰 강제 + 표준 체크리스트
> **사용 에이전트**: T13 Code Reviewer

## 다중 리뷰 원칙

> **한 번 보고 끝내지 않는다.**
> 모든 PR/diff는 **3차에 걸쳐** 리뷰. 각 차수에서 발견한 이슈를 별도로 기록.

```
1차 (구조)     → 2차 (안전성)    → 3차 (테스트 가능성)
       ↓               ↓                  ↓
   모든 차수 통과 시에만 사인오프
```

## 1차 리뷰: 구조와 가독성

### 책임 단일성 (SRP)
- [ ] 한 함수가 한 가지 일만 하는가?
- [ ] 200줄 넘는 파일은 분리 가능한가?
- [ ] 도메인 경계 침범 없음 (`lib/types.ts` 외 임의 인터페이스)

### 추상화
- [ ] 중복 코드 (DRY 위반) 없음
- [ ] 너무 깊은 중첩 (3단 이상) 없음
- [ ] 너무 많은 매개변수 (5개 이상) 없음

### 네이밍
- [ ] 함수/변수 이름이 의도를 드러내는가?
- [ ] `data`, `info`, `obj`, `temp` 같은 모호한 이름 없음
- [ ] 한글 도메인 용어 ↔ 영어 코드 용어 일관성

### 매직 넘버/문자열
- [ ] 의미 있는 상수/enum으로 추출
- [ ] 디자인 토큰 사용 (`#FF6B47` 같은 hex 없음)

## 2차 리뷰: 안전성과 보안

### 입력 검증
- [ ] API 입력에 Zod 스키마 적용
- [ ] 외부 데이터 (API 응답, DB 쿼리) 타입 가드
- [ ] null/undefined 분기 처리

### 에러 처리
- [ ] try/catch가 에러를 삼키지 않음
- [ ] 에러 메시지가 디버깅에 충분한 정보 포함
- [ ] 사용자 노출 메시지는 안전 (스택 트레이스 노출 X)
- [ ] 외부 API 실패 시 fallback

### 동시성
- [ ] race condition 분석
- [ ] 트랜잭션 경계 명확
- [ ] 멱등성 (재시도 안전)

### 보안 (OWASP)
- [ ] SQL Injection 방지 (Prisma 사용 → 자동)
- [ ] XSS 방지 (React 자동, but `dangerouslySetInnerHTML` 검토)
- [ ] CSRF 방지 (Next.js Server Actions / SameSite 쿠키)
- [ ] 비밀 정보 하드코딩 없음
- [ ] 인증/인가 체크 (보호 라우트)

### TravelDiary 특화
- [ ] **감사 로그** 추가됨 (S-13)
- [ ] **CTO 게이트** 통과 여부 (스택 변경 시) (S-18)
- [ ] AI 추천 데이터 진위 검증 호출 (T4)
- [ ] 위치 데이터 프라이버시 (T16)

## 3차 리뷰: 테스트 가능성과 도메인 일관성

### 테스트 가능성
- [ ] 외부 API 호출이 모킹 가능한 레이어로 분리
- [ ] 순수 함수 우선 (사이드이펙트 분리)
- [ ] 의존성 주입 가능 (전역 상태 최소화)
- [ ] 테스트 코드 함께 추가됨 (S-14)

### 도메인 일관성
- [ ] `lib/types.ts`의 타입을 그대로 사용
- [ ] 데이터 모델 (DAG) 가정 위반 없음
- [ ] 기존 패턴과 일관 (S-01~S-19 위반 없음)

### 디자인 시스템 (UI 변경 시)
- [ ] 디자인 토큰만 사용 (S-12)
- [ ] 기존 컴포넌트 재사용
- [ ] A11y 속성 (aria-*, alt) 적절
- [ ] 화면 LEVEL 정성도 적절

### 성능
- [ ] N+1 쿼리 없음
- [ ] 불필요한 리렌더 없음 (memo, useMemo 적절)
- [ ] 캐시 전략 (Redis/메모리/HTTP)
- [ ] 번들 크기 영향 (큰 라이브러리 추가 시 → CTO)

## 리뷰 산출물 형식

```markdown
## Code Review — [feature/branch-name]

**Reviewer**: T13
**Date**: 2026-04-29
**Scope**: 4 files, 350 LOC

### 1차 리뷰 (구조)
- ✅ SRP: `gatherEvidence()`가 단일 책임
- ⚠️ `EvidencePanel.tsx` 220줄 → `EvidenceSourceList` 추출 권장
- ✅ 네이밍: 명확

### 2차 리뷰 (안전성)
- ❌ 외부 API 실패 시 fallback 없음 (`fetchNaverReviews` throws)
- ✅ 입력 검증: Zod 적용
- ✅ 감사 로그: writeAuditLog 호출됨
- ⚠️ 사용자 IP 로깅 → 해시 처리 필요

### 3차 리뷰 (테스트)
- ❌ Naver API mock 부재 → 단위 테스트 불가능
- ✅ 순수 함수 분리 잘됨
- ⚠️ 테스트 추가 필요 (S-14 P0)

### 결정
- ⛔ 반려 (블로킹 2건, 권고 2건)
- 수정 후 재리뷰 예정

### 액션 아이템
1. [P0] fallback 추가
2. [P0] 테스트 추가
3. [P1] 컴포넌트 분리
4. [P1] IP 해시
```

## 차수별 통과 정책

| 차수 | 통과 기준 |
|------|----------|
| 1차 | ⚠️ 권고 허용, ❌ 블로커 0 |
| 2차 | ❌ 블로커 0 |
| 3차 | ❌ 블로커 0, 테스트 추가됨 |

블로커 1개라도 있으면 **반려 → 구현자 수정 → 재리뷰** 사이클.

## 리뷰 후속 단계

```
1차 통과 → 2차 시작
2차 통과 → 3차 시작
3차 통과 → T12 QA Lead 인계
T12 통과 → R1 CTO 사인오프 요청
```

## 리뷰어 자기 점검

리뷰어가 자기 자신을 의심:

- [ ] 같은 코드를 두 번 봤는가?
- [ ] 의심스러운 부분에 대해 직접 실행해 봤는가?
- [ ] 작성자에게 "왜 이렇게 했는지" 묻지 않고 추측하지는 않았는가?
- [ ] 큰 그림과 작은 디테일을 모두 봤는가?
