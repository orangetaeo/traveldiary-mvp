## 합의

### A1. 데이터 흐름
- 일정 데이터: 서버 컴포넌트(`app/itinerary/[id]/page.tsx`)에서 시드 import → `<ItineraryView>` 클라이언트 컴포넌트에 전달.
- `<ItineraryView>`가 `useState<ItineraryItem[]>` 보유. Replan 적용 시 새 일정으로 setState. 새로고침 시 시드로 리셋.

### A2. lib/replan.ts (순수 함수)
- `calculateAffectedRange(items, changedItemId)` — DFS로 의존성 후행 노드 수집.
- `generateReplanOptions(items, trigger)` — 3옵션 (추천/안전/강행) 생성.
- 각 옵션의 `impacts`는 `{ key, value, tone }` 배열 → ImpactDisplay에 전달.
- 알고리즘 정책 (S-06):
  - **추천**: flexible 항목만 트리거 분량 시프트, booked·fixed 보호. 후행 효과 최소화.
  - **안전**: 트리거 + 후행 모든 flexible 항목에 buffer 추가 (예약 안전).
  - **강행**: 모든 후행을 트리거 분량 그대로 시프트, booked가 영향받으면 경고 메시지 동봉.

### A4. 트리거 시연 시나리오 (T12 골든패스)
- 트리거: **Day 3 사오비치(pq-item-6) 90분 지연**.
- 영향 범위: pq-item-7(분짜) → pq-item-8(케이블카, booked) → pq-item-9(호국사).
- 추천: 분짜 +30분, 케이블카 그대로(booked 보호), 호국사 +30분 → impact = neutral.
- 안전: 분짜 +60분, 케이블카 buffer 30분 추가(예약 변경 필요 경고), 호국사 +60분 → impact = negative.
- 강행: 분짜 +90분, 케이블카 +90분(booked 충돌 ⚠️), 호국사 +90분 → impact = negative + warning.
