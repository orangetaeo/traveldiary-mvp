# 사이클 6 회고 (T18) — 견적 기획 사이클

## Keep

1. **사용자 피드백 즉각 반영** — "단조롭다 9가지 빠짐" → 사이클 6 분리 + 코드 0줄 합의 사이클로 진행. 매직 모먼트 추가 빌드를 무리하게 시도하지 않음.
2. **시드니 사이트 = 무료 골든셋** — 사용자 본인이 손으로 만든 정밀 가이드를 패턴으로 추출해 우리 v2 갭을 객관적으로 측정. 추측이 아닌 증거 기반.
3. **AskUserQuestion 4+3 분할** — 7개를 한 번에 묻지 않고 핵심 4개 → 후속 3개로 나눠 사용자 인지 부담 감소.
4. **사용자 결정 표 형태로 기록** — vision-v2.md 상단에 7개 결정 명시. 향후 사이클에서 "왜 이렇게 정했나"가 즉시 조회됨.

## Problem

1. **MVP 초기 v1 vision이 매직 모먼트 4개에 너무 좁게 집중** — 여행 동반자 기본기(지도·이미지·응급·체크리스트)가 무엇인지 사이클 0에서 정의했어야 함.
2. **시드니 사이트 인벤토리를 사이클 0에 했어야** — 사용자가 가진 정밀 패턴 자산을 미리 알았다면 사이클 1~5의 결정이 달랐을 가능성.
3. **데이터 모델이 사이클 1에 너무 단순화됨** — `bookingStatus` 3필드는 시드니 15+ 필드 대비 매우 빈약. 사이클 7+에서 큰 마이그레이션 부담.

## Try

1. **새 프로젝트 시작 시 "참고 사이트 인벤토리"를 사이클 0 의무 작업으로** — `_shared/agents/process/`에 work-process 갱신 후보.
2. **매직 모먼트 정의 시 "기본기 체크리스트" 통과 의무** — 지도·이미지·응급·공유·검색이 빠지면 매직 모먼트가 아닌 데모일 뿐.
3. **데이터 모델 v0에 "정밀 시나리오 1건"으로 검증** — 시드니의 한 일정 항목을 우리 모델에 매핑해 본 후 빈 필드 확인.

## 새 패턴 (도서관 갱신 후보)

### "참고 사이트 인벤토리 패턴" (process 갱신 후보)
- 사용자가 보유한 기존 사이트·도구를 사이클 0에 인벤토리화.
- WebFetch + 메인+서브 5~10개 병렬.
- 결과를 `docs/<source>-pattern-inventory.md`로 저장.
- 우리 MVP의 갭 분석 베이스라인.

### "Tier 분류 매트릭스" (T8 신규 스킬 후보)
- 갭 항목 → A~F 카테고리 → Tier 1/2/3 분류.
- 사용자 결정 시 한 매트릭스에서 통째로 보고 결정.
- vision v2 패턴.
