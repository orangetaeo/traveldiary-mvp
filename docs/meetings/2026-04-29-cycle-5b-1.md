### A3. Server Action 패턴 (R4 + T13)
```typescript
"use server";

export async function createTripFromOnboarding(input: CreateTripInput) {
  // demo trip은 시드로 분기 — sit case 아님
  if (!prisma) {
    return { id: DEMO_TRIP_ID, demo: true };
  }

  const trip = await prisma.$transaction(async (tx) => {
    const created = await tx.trip.create({ data: { /* User-less for now */ } });
    await tx.itineraryItem.createMany({ data: phuQuocItinerary.map(...) });
    return created;
  });

  await writeAuditLog({
    actorId: null,
    action: "trip.create",
    resource: "Trip",
    resourceId: trip.id,
    after: trip,
    metadata: { source: "onboarding", itemCount: 12, demoSeed: "phu-quoc" },
  });

  return { id: trip.id, demo: false };
}
```

### A4. 임시 정책: 사용자 없음
- 사이클 5b-1엔 인증 미도입. `Trip.ownerId`는 임시 시스템 유저(`SYSTEM_OWNER_ID`)로 채움.
- 마이그레이션 시 시스템 유저 1건 자동 생성.
- 카카오 OAuth(사이클 11)에서 실제 유저로 swap.

### A5. 시드 데이터를 DB로?
- **하지 않음**. 시드(`lib/seed/phu-quoc.ts`)는 영구 시연용. 사용자가 만드는 Trip은 시드와 분리.
- `getDemoTrip("demo-trip-phu-quoc")`는 항상 메모리 시드 반환 (URL 회귀 보호).

### A6. start 명령 정책
- `package.json`: `"start": "prisma migrate deploy && next start"`
- 빈 DATABASE_URL 시 prisma migrate deploy가 즉시 실패. 데모 모드는 build 시점이라 무관.
- Railway는 환경변수 주입 후 start이라 OK.
- 로컬에서 `npm start`를 DATABASE_URL 없이 돌리면 실패 — 의도된 동작 (production용).

### A7. 검증 시나리오
1. **데모 회귀**: 라이브 URL의 `/itinerary/demo-trip-phu-quoc` 그대로 동작 (시드 fallback)
2. **신규 Trip**: 온보딩 끝 → Server Action 호출 → DB에 Trip + 12 ItineraryItems + AuditLog 적재 → 새 ID로 redirect → `/itinerary/<new-id>` 정상 표시
3. **AuditLog**: PostgreSQL `SELECT * FROM "AuditLog" WHERE action='trip.create'` 1건 확인 (사용자 액션 후)
4. **Healthcheck**: `/api/health` `database: "ok"`
